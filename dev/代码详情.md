## 2. AI 亮点一：声明式 AI 编排 + 流式响应

### 2.1 为什么用这个技术？

- 这个项目不是“模型聊天 Demo”，而是一个**企业 ERP 智能助手后端**。如果把模型调用、记忆读取、RAG 检索、工具调用全写在 Controller 里，代码会很快失控。
- `@AiService` 的价值是把 **模型接入、系统提示词、记忆、RAG、Tool** 收敛到一个接口上，面试时可以一句话说明：  
  **“我没有把大模型当 HTTP Client 去调用，而是把它抽象成一个受控的业务编排层。”**
- 为什么要流式返回：ERP 问答常常会触发检索和 Tool，用户等待时间不可控。SSE 能把“等待黑盒”改成“边生成边可见”，提升交互确定性。

### 2.2 技术原理浅入深出

这一层本质是在做一件事：

1. 先把用户输入包装成一个**标准化 AI 请求**。
2. 再把系统提示词、历史消息、知识库证据、业务 Tool 一起注入到同一个推理上下文。
3. 最后通过流式 Token 输出把结果逐段发给前端。

如果用面试官容易理解的话说：

- **`@AiService` = 一个声明式的 AI Agent 装配器**
- **`MemoryId` = 会话上下文主键**
- **`SystemMessage` = 业务规则和边界条件**
- **`Tool` = 可被模型调用的业务函数**
- **SSE = 输出层的“增量传输协议”**

这套方案的关键，不是“能接模型”，而是“**能稳定接进业务**”。

### 2.3 代码级实现细节

核心入口在 `ConsultantService`：

```java
@AiService(
    wiringMode = AiServiceWiringMode.EXPLICIT,
    chatModel = "openAiChatModel",
    streamingChatModel = "openAiStreamingChatModel",
    chatMemoryProvider = "chatMemoryProvider",
    contentRetriever = "contentRetriever",
    tools = {
        "systemManagementTool",
        "materialTool",
        "inventoryBillTool",
        "financeTool",
        "reportTool"
    }
)
public interface ConsultantService {

    @SystemMessage(fromResource = "system.txt")
    TokenStream chat(@MemoryId String memoryId, @UserMessage String input);
}
```

逐行怎么理解：

- `wiringMode = EXPLICIT`：不用自动猜 Bean，而是**显式绑定**模型、记忆、RAG、Tool，避免装配歧义。
- `chatModel` / `streamingChatModel`：把**普通推理**和**流式推理**都接进来。
- `chatMemoryProvider`：每次请求都会按 `memoryId` 自动挂接历史消息。
- `contentRetriever`：模型需要证据时，会调用 RAG 检索器。
- `tools`：模型可以把自然语言意图转成业务调用。
- `@SystemMessage(fromResource = "system.txt")`：系统行为规则不写死在 Java 里，而是抽到提示词文件。

`ChatController` 的职责非常轻：

```java
@GetMapping(value = "/chat", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<String> chat(@RequestParam String memoryId, @RequestParam String message) {
    return Flux.create(
        emitter -> aiChatObservationService.streamChat(memoryId, message, emitter),
        FluxSink.OverflowStrategy.BUFFER
    );
}
```

这里的设计重点：

- Controller 不直接调模型 SDK。
- 它只负责把请求交给 `AiChatObservationService`。
- 输出协议直接是 `TEXT_EVENT_STREAM`，即 SSE。

### 2.4 从请求到响应的数据流

```text
前端 /chat
 -> ChatController
 -> AiChatObservationService
 -> ConsultantService.chat(memoryId, message)
 -> 读取 system.txt
 -> 挂接 memoryId 对应会话记忆
 -> 必要时执行 RAG 检索
 -> 必要时执行 Tool
 -> streamingChatModel 增量返回
 -> SSE 推回前端
```

### 2.5 踩坑与优化

下面这 3 个问题，都是**从当前代码结构可以直接反推出已经处理过的坑**：

1. **如果把 AI 编排写在 Controller 里，业务膨胀非常快**
   - 解决：抽成 `@AiService + service orchestrator`，Controller 只保留输入输出边界。
2. **流式返回过程里，成功回调和错误回调可能重复收尾**
   - 解决：`AiChatObservationService` 用 `AtomicBoolean finalized` 做幂等收尾。
3. **AI 能力强，但没有日志和 trace，就无法排查线上问题**
   - 解决：所有 `/chat` 请求统一走 `AiChatObservationService`，在这里串起 partial response、retrieval、tool、complete、error 全链路采样。

### 2.6 性能量化指标

这部分面试时建议坚持“**代码可证，不瞎报收益**”：

- **流式返回协议**：SSE，即用户不是等完整答案后再看，而是边生成边收到。
- **聊天模型超时**：`60s`
- **聊天模型最大重试**：`3`
- **Embedding 模型最大重试**：`5`
- **Embedding 批处理上限**：`10` 段/批
- **慢请求阈值**：`5000ms`
- **监控指标**：`ai_chat_requests_total`、`ai_chat_latency`、`ai_chat_first_token_latency`

建议面试表达：

> “这个项目的性能指标我不乱报，我讲代码里真实存在的控制阈值和观测指标，比如首字延迟、总延迟、慢请求阈值 5 秒，以及流式输出带来的主观等待优化。”

### 2.7 面试官高频追问模拟（Q&A）

**Q1：为什么不用自己封装 OpenAI SDK，而要用 LangChain4j？**  
A：因为这个项目不是只调用一个模型接口，而是要同时编排 **提示词、记忆、RAG、Tool、流式输出**。如果手写 SDK，很快会变成一堆 if/else 和回调拼装；`@AiService` 让这些依赖显式声明，维护成本更低。

**Q2：如果以后模型服务商变了，改造成本大吗？**  
A：不大。项目接的是 **OpenAI 兼容协议**，并没有把业务代码绑死在某个专有 SDK 上。改动主要在 `application.yml` 的 `base-url / api-key / model-name`。

**Q3：为什么要做流式返回？**  
A：因为这个系统经常不是单轮纯文本回复，可能会触发 RAG 和 Tool。非流式下用户只能等待；流式下至少能看到系统在持续工作，用户感知更稳定，也更适合企业场景演示。

## 3. AI 亮点二：企业级 RAG 检索，而不是“搜到就答”

### 3.1 为什么用这个技术？

- 这个 AI 助手要回答大量 **ERP 操作说明、业务流程、系统手册** 类问题，仅靠通用大模型会出现两个问题：
  - 它知道“通用 ERP 常识”，但**不知道你这套系统怎么设计的**。
  - 它会给出“看起来合理”的答案，但未必来自项目知识库。
- 所以这里必须引入 RAG，让回答建立在**项目内真实文档证据**上。
- 但简单 RAG 也不够，因为“搜到一点像的文本”并不等于“足够支撑确定性回答”。所以项目又加了 `StrictContentRetriever` 这一层做**证据门控**。

### 3.2 技术原理浅入深出

这一层可以概括成 4 步：

1. **文档切块**：把 PDF 手册拆成适合向量检索的片段。
2. **向量化**：把片段转成 embedding，写入 Redis 向量库。
3. **检索**：把用户问题也转成 embedding，做相似度搜索。
4. **门控**：不是有结果就给模型，而是只把**长度合格、分数够高**的证据返回。

这里最值得讲的不是“做了 RAG”，而是“**做了拒答机制**”：

- `minScore = 0.68`：低于这个分数，直接不进候选。
- `answerableMinScore = 0.74`：就算检索到了，如果最高分没过 0.74，也会判定为**证据不足**。
- `minSegmentLength = 40`：太短的片段认为是噪声。

面试官通常会喜欢这个点，因为它说明你在做的是**可信问答**，不是“尽量回答”。

### 3.3 代码级实现细节

`StrictContentRetriever` 的关键逻辑可以概括成：

```java
Response<Embedding> response = embeddingModel.embed(query.text());

EmbeddingSearchRequest request = new EmbeddingSearchRequest(
    response.content(),
    aiRagProperties.getRetrieval().getMaxResults(),
    aiRagProperties.getRetrieval().getMinScore(),
    null
);

EmbeddingSearchResult<TextSegment> searchResult = embeddingStore.search(request);

for (EmbeddingMatch<TextSegment> match : rawMatches) {
    if (!isUsable(match)) {
        incrementRejection("short_segment");
        continue;
    }
    acceptedMatches.add(match);
}

if (acceptedMatches.isEmpty()) {
    return List.of();
}

if (topScore < aiRagProperties.getRetrieval().getAnswerableMinScore()) {
    incrementRejection("low_confidence");
    return List.of();
}
```

逐段解释：

- 先对**用户问题做 embedding**。
- 用 `maxResults` 和 `minScore` 去 Redis 向量库检索。
- 命中的每条结果先过 `isUsable()`：
  - 片段不能是空的；
  - 文本长度不能低于 `minSegmentLength`。
- 如果最高分不够高，即使有结果，也要**拒答**。

这就是项目里最有分量的 RAG 设计点：

> **RAG 不是把检索到的文本硬塞给模型，而是先判断这份证据值不值得信。**

### 3.4 文档入库链路

文档不是一次性暴力重建，而是做了**增量同步**：

```java
String docId = sha256(filename.toLowerCase(Locale.ROOT).getBytes(UTF_8));
String docSha256 = sha256(bytes);

String previousSha256 = asString(hashOperations.get(manifestKey, docId));
if (Objects.equals(previousSha256, docSha256)) {
    // 文档没变，跳过
    continue;
}

if (StringUtils.hasText(previousSha256)) {
    removeDocument(docId);
}

List<TextSegment> segments = pdfKnowledgeDocumentLoader.load(bytes, docId, filename, docSha256);
Response<List<Embedding>> response = embeddingModel.embedAll(segments);
embeddingStore.addAll(ids, response.content(), segments);
hashOperations.put(manifestKey, docId, docSha256);
```

这里的工程价值非常高：

- `docId`：标识“这是哪份文档”
- `docSha256`：标识“这份文档内容有没有变”
- 没变化就**跳过重建**
- 变化了就**删旧向量、写新向量**
- 资源目录删掉的文档，也会**清理失效索引**

### 3.5 PDF 分块策略

这部分参数也很适合面试时拿来讲细节：

- `pageWindow = 1`
- `maxSegmentSize = 350`
- `overlap = 60`
- `minTextLength = 80`
- `renderDpi = 200`
- `ocrEnabled = false`
- `ocrLanguage = chi_sim+eng`

如果面试官问“为什么 350 和 60”，建议回答：

- 片段太长，语义会被稀释；
- 片段太短，信息不完整；
- `overlap` 用来缓解“关键句恰好被切断”的问题。

### 3.6 踩坑与优化

1. **PDF 原始文本噪声很大**
   - 解决：先清洗页面文本，再做递归分块，而不是直接拿整页文本入库。
2. **简单相似度检索容易把低质量片段喂给模型**
   - 解决：增加 `minSegmentLength` 和 `answerableMinScore` 两层门槛。
3. **知识库更新后，旧向量残留会污染检索**
   - 解决：引入 `manifestKey + docSha256`，支持增量同步与失效清理。

### 3.7 性能量化指标

这部分有非常多**代码可证数据**：

- 检索候选上限：**4**
- 召回最低阈值：**0.68**
- 可回答最低阈值：**0.74**
- 最短有效分片：**40 字符**
- 文本分块大小：**350**
- 分块重叠：**60**
- OCR 开关：默认 **false**
- OCR 最低文本长度触发条件：**80**
- Embedding 批量大小：**10**
- RAG 监控指标：`ai_rag_hits_count`、`ai_rag_top_score`、`ai_rag_search_latency`、`ai_rag_rejections_total`

建议面试表达：

> “我不直接报‘准确率提升 20%’这种没证据的数据，我会讲系统里已经落地的检索阈值、分块参数、拒答条件和监控指标，因为这些是可复现、可证明的。”

### 3.8 面试官高频追问模拟（Q&A）

**Q1：如果知识库规模暴增 100 倍，这套方案怎么扩？**  
A：先拆成两层回答。短期内可以通过优化 Redis 索引、分批 embedding、异步同步和更严格 metadata 过滤提升可用性；长期看会把向量库从单机 Redis 演进到更适合大规模向量检索的专用存储，或者按租户/业务域拆索引。

**Q2：为什么不检索到内容就直接回答？**  
A：因为企业场景的核心问题不是“有没有回答”，而是“回答能不能被证据支撑”。所以这里做了两层门控：低于 `minScore` 的结果不进候选，最高分低于 `answerableMinScore` 的直接拒答。

**Q3：如果 PDF 是扫描件怎么办？**  
A：代码里已经预留了 `OCR` 能力，技术栈是 `PDFBox + Tess4J`，只是当前配置 `ocrEnabled=false`。也就是说，这不是能力缺失，而是当前环境出于成本和稳定性考虑默认关闭。

## 4. AI 亮点三：多租户会话记忆，不让 AI 串话

### 4.1 为什么用这个技术？

- 企业 AI 场景不是单用户单会话，而是**多租户、多账号、多会话并发**。
- 如果聊天记忆只按 `memoryId` 存，最容易出现两个高风险问题：
  - 同名会话 ID 跨租户串数据
  - 同一租户不同用户共用上下文
- 所以这个项目没有把记忆当简单缓存，而是把它当**带隔离语义的会话状态存储**。

### 4.2 技术原理浅入深出

会话记忆这里做了两层拆分：

1. **Redis 短期记忆**
   - 给模型做上下文；
   - 有 TTL；
   - 只保留最近 N 条消息。
2. **MySQL 长期归档**
   - 给前端会话历史查看；
   - 用于审计和历史追溯；
   - 不直接参与模型推理。

也就是说，这个项目把“**模型需要的上下文**”和“**业务需要的历史记录**”拆开了。

### 4.3 代码级实现细节

`CommonConfig` 里配置了窗口记忆：

```java
public ChatMemoryProvider chatMemoryProvider(ChatMemoryStore redisChatMemoryStore,
                                             AiMemoryProperties aiMemoryProperties) {
    return memoryId -> MessageWindowChatMemory.builder()
            .maxMessages(aiMemoryProperties.getMaxMessages())
            .id(memoryId)
            .chatMemoryStore(redisChatMemoryStore)
            .build();
}
```

这里的重点：

- `MessageWindowChatMemory`：不是无限历史，而是窗口记忆。
- `maxMessages` 当前配置为 **14**。
- 实际落盘由 `RedisChatMemoryStore` 完成。

Redis Key 的设计非常值得讲：

```java
String key = aiMemoryProperties.getKeyPrefix()
        + ":tenant:" + tenantPart
        + ":user:" + userPart
        + ":memory:" + resolvedMemoryId;
```

最终长这样：

```text
erp:chat-memory:tenant:{tenantId}:user:{userId}:memory:{memoryId}
```

面试时一定要主动说这个点，因为它直接体现了你的**安全边界意识**。

### 4.4 多租户上下文是怎么解析的？

`TenantRequestInterceptor` 负责把请求头中的用户信息转成租户和角色上下文：

```java
Long tenantId = resolveTenantId(request);
TenantContextHolder.setTenantId(tenantId);
UserContextHolder.setCurrentUser(resolveCurrentUser(request, tenantId));
```

`resolveCurrentUser()` 还会把数据库中的角色名转成 `Set<Role>`，后续 Tool 鉴权直接复用。

也就是说：

- HTTP 请求进入系统时先解析用户和租户；
- 后续记忆、Tool、日志都从这个上下文拿租户和用户；
- AI 层没有自己重复解析用户，而是复用统一上下文。

### 4.5 踩坑与优化

1. **会话记忆只按 memoryId 组织，会导致串话**
   - 解决：Key 加入 `tenantId + userId + memoryId` 三维隔离。
2. **无限历史会推高 token 成本**
   - 解决：只保留最近 **14** 条消息，形成窗口记忆。
3. **Redis 不可用时，聊天服务可能整体不可用**
   - 解决：当前代码已经把 Redis 读写改成容错；读取失败时返回空上下文，写入失败时只是不持久化，不再直接打崩主链路。

### 4.6 性能量化指标

- 会话窗口：**14 条消息**
- Redis TTL：**86400 秒**
- Key 维度：**tenant + user + memory**
- 单次获取失败降级行为：**返回空消息列表**

建议面试表达：

> “这层的核心指标不是吞吐量，而是隔离粒度和 token 控制。我把窗口固定为 14 条，并把租户和用户维度编码进会话 Key，优先保证数据不串和上下文不过度膨胀。”

### 4.7 面试官高频追问模拟（Q&A）

**Q1：为什么不把所有历史都给模型？**  
A：因为历史越长，token 成本越高，而且噪声会越来越大。窗口记忆能兼顾上下文连续性和成本控制。

**Q2：为什么 Redis 记忆和 MySQL 归档分开？**  
A：因为它们的目标不一样。Redis 是高频、短期、给模型用；MySQL 是长期、结构化、给前端和审计用。混在一起会导致查询模型和业务模型都变重。

**Q3：如果两个用户同时用同一个 memoryId，会不会串？**  
A：不会，因为真正的 Redis Key 不是裸 `memoryId`，而是 `tenant:user:memory` 三维组合。

## 5. AI 亮点四：Tool 执行型助手，把 AI 接到 ERP 真业务

### 5.1 为什么用这个技术？

- 面试里一定要强调：这个项目不是“AI 给建议”，而是“AI 能执行业务动作”。
- 单纯聊天机器人最大的瓶颈是：它可以解释流程，但**不能真的帮你做事**。
- Tool 机制的意义，就是把自然语言请求映射成真实的 ERP Service 调用。

当前接入主 AI 服务的 Tool 有：

- `systemManagementTool`
- `materialTool`
- `inventoryBillTool`
- `financeTool`
- `reportTool`

这意味着模型不仅能答“怎么做”，还能查数据、建商品、开单、记账、拉报表。

### 5.2 技术原理浅入深出

Tool 本质上是把 AI 输出从“自然语言”变成“结构化函数调用”：

1. 用户说自然语言。
2. 模型识别意图。
3. 模型选择对应 Tool，并构造参数。
4. Tool 调用 Java 方法。
5. Java 方法下沉到 Service/Mapper，触发真实业务逻辑。

如果面试官问“这和函数调用有什么区别”，可以答：

> “本质上就是函数调用，但这里接的是企业业务能力，所以我还叠加了鉴权、监控和 trace 采集，不是裸函数。”

### 5.3 代码级实现细节

AI Tool 不是直接暴露 DAO，而是明确分层：

```java
@Tool("创建系统消息")
@RequireRole({Role.TEACHER, Role.ADMIN})
public ToolActionResult createMessage(String title, String content, Long userId, String type) {
    return systemManagementService.createMessage(title, content, userId, type, null);
}
```

这个设计说明了三件事：

- Tool 是**AI 可调用入口**，不是业务逻辑本体。
- 业务逻辑仍然在 `service` 层。
- Tool 级别就挂了 `@RequireRole`，说明你把 AI 当作一个**需要权限控制的执行入口**。

### 5.4 Tool 监控与权限控制

`ToolExecutionMonitoringAspect` 会统一给 Tool 打点：

```java
@Around("within(com.example.consultant.tools..*) && @annotation(dev.langchain4j.agent.tool.Tool)")
public Object monitorTool(ProceedingJoinPoint joinPoint) throws Throwable {
    String toolName = ((MethodSignature) joinPoint.getSignature()).getMethod().getName();
    Timer.Sample sample = Timer.start(meterRegistry);
    try {
        Object result = joinPoint.proceed();
        Counter.builder("ai_tool_invocations_total")
            .tag("tool", toolName)
            .tag("status", "success")
            .register(meterRegistry)
            .increment();
        return result;
    } finally {
        sample.stop(timer);
    }
}
```

`ToolRoleAuthorizationAspect` 会在 Tool 执行前做角色校验：

```java
RequireRole requireRole = resolveRequireRole(method, joinPoint.getTarget().getClass());
rolePermissionService.assertHasAnyRole(resourceName, requireRole.value());
```

这两层的组合非常重要：

- **监控**：知道模型都调用了什么 Tool、耗时多少。
- **鉴权**：知道模型“能不能替这个用户做这件事”。

### 5.5 踩坑与优化

1. **AI 能调 Tool，但如果没有角色隔离，会越权操作**
   - 解决：加 `@RequireRole + AOP`，在 Tool 层统一拦截。
2. **Tool 失败后，如果只返回异常，不知道模型具体做错了什么**
   - 解决：`AiChatObservationContext` 记录 `toolName / argumentsJson / resultPreview / success / errorMessage`。
3. **Tool 看似成功，但结果文本里其实有失败信号**
   - 解决：在线评估阶段再扫一次 `resultPreview`，识别 suspicious/failure markers。

### 5.6 性能量化指标

- 当前接入主 AI 服务的 Tool：**5 个**
- Tool 级监控指标：
  - `ai_tool_invocations_total`
  - `ai_tool_latency`
- 请求级 Tool 留痕字段：
  - `tool_name`
  - `arguments_json`
  - `result_preview`
  - `success`
  - `error_message`
  - `sequence_no`

建议面试表达：

> “我不是只做了 Tool 接入，而是把 Tool 当成一个企业动作执行面，额外补了权限、监控和审计，这一点和普通函数调用 Demo 不一样。”

### 5.7 面试官高频追问模拟（Q&A）

**Q1：模型把参数传错了怎么办？**  
A：第一层靠系统提示词和 Tool 定义约束参数结构，第二层靠 Service 层继续做业务校验，第三层把错误结果回写到 Tool trace，便于诊断。

**Q2：为什么权限控制放在 Tool 层，而不是只放在 API 层？**  
A：因为 Tool 已经是 AI 的执行边界。如果只在 API 层做控制，AI 内部仍然可能越权访问业务方法；放在 Tool 层可以保证“模型能调用的动作”和“用户有权限的动作”是同一边界。

**Q3：如果以后 Tool 数量翻倍，怎么管控复杂度？**  
A：会按业务域继续拆 Tool，比如主数据、库存、财务、报表，每个 Tool 保持薄入口，复杂逻辑继续留在 Service。再加上 AOP 统一做权限和监控，就不会因为 Tool 多而难维护。

## 6. AI 亮点五：可观测性 + 在线评估，让 AI 可运营、可排障

### 6.1 为什么用这个技术？

- AI 项目最难的不是“跑起来”，而是“**出了问题怎么定位**”。
- 传统业务接口看 HTTP 状态码就够了，但 AI 链路里还要知道：
  - 首字延迟高不高；
  - 检索有没有命中；
  - Tool 有没有失败；
  - 回答是不是低置信还在硬答；
  - 本轮整体应该判成 PASS/WARN/FAIL。
- 所以这里做了两层能力：
  - **请求级 trace**
  - **规则化在线评估**

### 6.2 技术原理浅入深出

整条链路里，`AiChatObservationService` 是总线：

1. 请求开始，创建 `AiChatObservationContext`
2. partial response 到来时，记录首字时间和输出长度
3. 检索回调时，记录证据快照
4. Tool 回调时，记录调用链明细
5. 请求结束后，落 MySQL，计算评分，发布 Micrometer 指标

你可以把它理解成：

> **它不是“记录聊天日志”，而是在给每一次 AI 请求生成一个可回放、可分析、可评分的运行画像。**

### 6.3 代码级实现细节

`AiChatObservationContext` 会累积整次请求状态：

```java
private final String requestId = UUID.randomUUID().toString().replace("-", "");
private final List<Map<String, Object>> retrievalSnapshots = new ArrayList<>();
private final List<AiToolTrace> toolTraces = new ArrayList<>();

private Long latencyMs;
private Long firstTokenLatencyMs;
private int streamedChars;
private int retrievedCount;
private Double topRetrievalScore;
private String retrievalRejectedReason;
private Integer inputTokens;
private Integer outputTokens;
private Integer totalTokens;
```

这意味着一次请求结束后，不只是知道“成没成功”，而是能知道：

- **慢在首字，还是慢在整体**
- **检索到了几条**
- **最高分多少**
- **为什么拒答**
- **调用了几个 Tool**
- **token 消耗大概是多少**

在线评估器 `AiResponseEvaluationService` 的评分权重也很清晰：

```java
int overallScore = (int) Math.round(
    executionScore * 0.35 +
    groundingScore * 0.30 +
    toolScore * 0.20 +
    responseScore * 0.15
);
```

四个维度分别看什么：

- `executionScore`：请求是否正常完成、是否空答、首字是否过慢
- `groundingScore`：回答是否被检索证据支撑
- `toolScore`：Tool 有没有失败或可疑结果
- `responseScore`：回答是不是过短、是不是明显兜底

### 6.4 SQL 持久化结构

这部分也很适合面试讲“我不是只打日志，我做了结构化留痕”：

- 主表：`ai_request_trace`
- 子表：`ai_tool_trace`

主表字段包括：

- `request_id`
- `memory_id`
- `tenant_id`
- `user_id`
- `question`
- `response`
- `status`
- `latency_ms`
- `first_token_latency_ms`
- `streamed_chars`
- `retrieved_count`
- `top_retrieval_score`
- `tool_call_count`
- `evaluation_score`
- `evaluation_status`
- `risk_level`
- `retrieval_snapshot_json`

这已经不是“日志字符串”，而是**可运营查询的数据模型**。

### 6.5 踩坑与优化

1. **检索回调和请求收尾回调时序不一致，可能导致 trace 丢证据**
   - 解决：增加 `AiObservationRegistry + AiRetrievalAuditService`，用短期缓存兜底。
2. **如果用第二个模型给 AI 质检，成本高且结论波动大**

3. **AI 响应太长会把 trace 表撑爆**

### 6.7 面试官高频追问模拟（Q&A）

**Q1：为什么在线评估不用 LLM-as-a-judge？**  
A：因为这个项目更强调稳定性和可解释性。规则评分器虽然没有模型法官那么灵活，但更便宜、更稳定，也能明确说明为什么是 WARN/FAIL。

**Q2：如果某次 AI 回答错了，你怎么排查？**  
A：先查 `request_trace` 看状态、首字延迟、总延迟、检索命中、最高分、风险等级；再查 `tool_trace` 看调用了哪个 Tool、参数是什么、返回结果有没有失败标记；必要时再看 `retrieval_snapshot_json` 回放证据。

**Q3：如果以后要做离线评测，你的现有设计能复用什么？**  
A：请求 trace、检索快照、Tool 留痕、评分结果都已经结构化了，离线评测可以直接基于这些数据做样本抽取和误差分析，不需要重新埋点。

这个项目现在的“意图识别”本质上不是单独一层分类模型，而是“Prompt 规则 + LLM 自主判断 + Tool/RAG 路由”的组合。

现在怎么做的

入口是 ConsultantService.java (line 10)，这里把 chatMemoryProvider、contentRetriever 和 5 个业务 Tool 一起挂到 @AiService 上。也就是说，模型在回答前就已经拿到了“可走知识库、可调工具”的能力集合。
真正的“意图规则”主要写在 system.txt (line 22) 里。它会先把用户请求粗分成几类：
查询/统计/查看/分析：优先走 Tool
新增/创建/保存/更新/开单/记账：优先走 Tool 执行
怎么做/有什么用/能做什么：优先走说明型回答
system.txt 还继续做了领域级映射，比如：
商品相关 -> materialTool
采购/销售/出入库 -> inventoryBillTool
财务单据 -> financeTool
报表/库存预警 -> reportTool
这部分在 system.txt (line 33) 和 system.txt (line 40)。
如果模型判断是“知识问答型”，就会走 contentRetriever；如果判断是“业务操作型”，就会走 Tool。这个调用链在 AiChatObservationService.java (line 64) 能看到：onRetrieved(...) 和 onToolExecuted(...) 都已经接进观测链路了。
另外，项目还在 system.txt 里做了很多参数追问模板，比如创建商品前必须先查分类、查单位，再确认 categoryId/unitId，这属于“意图识别后的参数补全”。
所以一句话总结当前方案

现在是**“大模型内隐式意图识别”**，不是显式 Intent Classifier。
优点是开发快、灵活、扩展 Tool 简单。
缺点是可控性、可解释性、可评估性还不够强。
现在方案的几个明显短板

没有单独的 intent 字段落库。现在只知道最终有没有调 Tool、有没有检索，但不知道模型一开始把这次请求判成了什么意图。
意图识别高度依赖 system.txt 和模型理解，稳定性受 Prompt 质量影响大。
没有“意图置信度”。比如一句“帮我看下采购情况”，到底是查单据、查统计、还是问流程，当前没有显式分数。
没有“多轮状态机”。比如用户先说“帮我开销售单”，下一轮补“客户是华东商贸”，目前更多靠会话记忆，不是严格表单式状态推进。
reservationTool 已实现，但还没挂进主 AI 服务，说明意图路由还没完全覆盖所有场景。
我建议的改进方案

加一层显式 Intent Router

在 /chat 前先做一次轻量意图分类，输出：
intentType: FAQ / KNOWLEDGE_QA / DATA_QUERY / ACTION / REPORT / AMBIGUOUS
domain: system / material / inventory / finance / report / reservation
confidence
这样可以先路由，再决定是走 RAG、Tool，还是普通对话。
做“规则优先，LLM 兜底”的混合识别

强关键词场景直接规则命中：
“创建/新增/开单/记账” -> ACTION
“统计/排行/预警/报表” -> REPORT
“怎么做/流程/作用” -> KNOWLEDGE_QA
模糊表达再交给 LLM 分类。
这样比纯 Prompt 更稳，也更省 token。
把意图识别结果结构化落库

在 ai_request_trace 里新增：
intent_type
intent_domain
intent_confidence
router_source(rule|llm|hybrid)
这样后面才能分析“哪些意图最常误判”。
把“参数补全”升级成状态机

尤其是开单、建商品、记账这类执行型场景。
不要只靠模型自由追问，应该维护一个 slot filling 状态：
已确认字段
缺失字段
